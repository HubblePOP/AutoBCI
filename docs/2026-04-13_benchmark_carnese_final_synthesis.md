# Benchmark Carnese：步态划分从零开始的框架性能测试

**最终综合方案（基于 Codex 与 Claude Code 评审意见的最终提炼）**
**日期：2026-04-13**

---

经过对同事们（Codex 与 Claude Code）前期意见的复盘，我们发现关于“如何做好本次 Benchmark”的争论，核心都指向了四个关键的思考问题。本文档按这四个问题进行重新梳理，并在最后给出了明确的执行决策建议。

## 思考问题一：本次 Benchmark 到底在测什么？（能力边界划分）

**共识：** 必须把“任务指标”和“框架指标”严格切分开，不能混为一谈。

*   **任务层（步态划分能不能做）：** 这是业务问题。即用运动学数据能否稳定切分出支撑期和摆动期。
*   **框架层（AutoResearch 的调度能力）：** 这是 Benchmark 的战略目标。即框架在拿到一个全新任务时，其**探索、试错、收敛、修复**的效率如何。

**结论：** 
如果把这两个层面的指标混在一个总分里，当效果不佳时，我们无法归因是“任务太难/定义有误”还是“Agent 调度算法太蠢”。因此，**任务的基础设施必须预先给好，Agent 只负责在这个确定的基础设施上做研究和调度。**

---

## 思考问题二：新任务（步态划分）的本质特征与裁判规则是什么？

**共识：** 步态划分不能照搬“运动想象（Motor Imagery）”那种有明确开始/结束提示音、固定窗口、单次离散分类的静态范式。

步态是一种**连续运动**，存在真实的神经传导与系统延迟，不能假设预测会与真实切换点做到毫秒级的完美对齐。因此，我们必须确立严格的裁判规则（Timing Policy）：

1.  **任务定性：** 连续运动上的阶段识别任务，允许固定的生理延迟，**坚决不允许事后按真值择时作弊**。
2.  **择时规则（Timing Policy）：**
    *   **允许：** 模型设计中包含一个固定的全局延迟 $\tau$（仅能在 Train/Val 集上确定并在 Test 时固定）。
    *   **不允许：** 逐条试次单独找最佳时移（Shift）、动态时间拉伸（Warping）、以及事后拿结果对齐真值再打分。
3.  **多维评分口径（分开报告，不合并）：**
    *   **试次可用率：** 标签定义本身的稳定性。
    *   **事件误差（ms）：** 切分点的绝对精度。
    *   **阶段重叠率（IoU）：** 整体区间的匹配度。
    *   **延迟分布：** 区分模型是“整体慢半拍”还是“根本没学会”。

---

## 思考问题三：当前框架如何在新任务上冷启动？（环境与规则隔离）

**共识：** 当前的 Agent 并不是从“完全空白”起跑的（它需要 track manifest、smoke/formal command）。且旧的 `CONSTITUTION.md` 锁死了旧任务，会导致 Agent 被拉回老路。

**结论：** 必须引入 **Benchmark Bootstrap 阶段**，并做彻底的沙盒隔离。

1.  **预写最小可运行环境（Task Pack）：** 人类必须预先写好评分脚本（`eval_gait_phase.py`）以及最小可运行的 track（保证有 smoke/formal 指令供 Agent 调用）。
2.  **起草独立的上位契约：** 编写 `docs/CONSTITUTION_BENCHMARK_GAIT_PHASE.md`，替换掉 canonical task 和 metric，让 Agent 明确当前唯一的主线是步态划分，不再被旧规则干扰。
3.  **隔离产物目录：** 所有的 ledger、状态和 review packets 必须写入独立的 `artifacts/gait_phase_benchmark/`。

## 思考问题四：如何科学地衡量框架的调度能力？

**共识：** 旧的 `--compare` 脚本直接读取连续回归分数（如 `val_r`）来定义突破，这和新任务的指标完全不兼容。

**结论：** 统一数据接口，剥离独立能力轴。

1.  **统一突破口径：** 评分脚本必须将“试次可用率”写入 ledger 的 `val_primary_metric` 字段。这样 `--compare` 脚本无需修改，对比的将是框架的**行为指标**（如：突破率、每次突破成本、最大连续干旱期、方向多样性等），而不是直接对比业务分数的绝对值。
2.  **独立报告能力轴：**
    *   **搜索利用率：** 联网搜索后产生代码修改的比例。
    *   **环境自修复：** 命令报错后 Agent 自行回滚并重试修复的次数。
    *   **自主换向：** 算法类别（Algorithm Family）主动切换的次数。
    这三项能力很重要，但不应混入总分，应单独作为分析雷达图的维度。

## 五、遗留决策与我的独立建议

针对前期讨论遗留的三个核心决策，我的最终判定意见如下：

### 决策 1：步态标签的参考"真值"怎么定？
*   **我的建议：A（用极值法预先生成一份参考标签作为全局真值）。**
*   **理由：** Benchmark 的第一要义是**基准的绝对稳定与可复现**。在第一版（V1）测试中引入人工标注或多方法共识，会带来极大的对齐成本与主观波动。极值法即使有例外，但它足够刚性，能够作为一个公平的靶子来测试 Agent 的射击能力。

### 决策 2：评分脚本是否作为 Task Pack 的一部分预写？
*   **我的建议：必须预写。**
*   **理由：** 裁判不能下场踢球。如果我们连评分脚本都让 Agent 自己写，那这就变成了“任务形式化能力”与“算法调度能力”的混合测试，一旦失败，根本无法归因。预写评分脚本是形成闭环（smoke/formal）的刚需底线。

### 决策 3：Benchmark 的运行时长怎么定？
*   **我的建议：A（16轮迭代）。**
*   **理由：** 作为初期测试，16 轮已经足够观察 Agent 在“探索（发散） -> 试错 -> 收敛”这三个阶段的典型行为特征。如果给的时间/轮次过长（如 24 轮或 8 小时），不仅浪费算力（烧 Token），反而会掩盖框架在“早期方向选择错误或陷入局部最优”这一关键缺陷。16 轮是一个极佳的压力测试边界。
