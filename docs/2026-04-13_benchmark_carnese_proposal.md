# Benchmark Carnese 方案

## 这份方案要回答的三个问题

这份方案不是在讨论“当前哪条模型分数更高”，而是在讨论：

1. 当我们的研究问题改了之后，要不要重定义 `program`。
2. 如果要，新的 `program` 应该定义什么。
3. 我们到底拿什么来衡量 AutoResearch 框架本身，而不是只看某个模型的最终分数。

当前结论先说在前面：

- **要定义新的 `program`。**
- 但**不是替换当前主线 `program`**。
- 而是新增一套**并行的 benchmark program**，专门用来衡量 AutoResearch 作为研究系统本身的能力。

换句话说：

- 现有 `program` 服务的是“连续脑电到运动学解码”这条正式研究主线。
- 新的 `Benchmark Carnese program` 服务的是“从零开始定义任务后，框架自己能不能发现方向、搭环境、搜索、换向、落实实验”的系统能力评估。

这两个 `program` 不是同一层东西，不能混成一份。

## 为什么现有 `program` 不够用

当前 [tools/autoresearch/program.md](/Users/mac/Code/AutoBci/tools/autoresearch/program.md) 和 [tools/autoresearch/program.current.md](/Users/mac/Code/AutoBci/tools/autoresearch/program.current.md) 的前提是：

- canonical 任务已经确定
- 数据划分已经确定
- 正式晋升门已经确定
- 当前 track 语义已经存在
- 系统主要是在既有研究问题里做局部推进

这套设计很适合当前主线，也就是：

- 固定任务
- 固定裁判
- 固定主线
- 在既有轨道里找更好的表征、模型和训练口径

但它不适合回答下面这个新问题：

> 如果只给系统一个任务定义，不给它历史最好结果、不提前写好轨道、不告诉它先试哪种模型，它自己能不能把一条新任务从零做起来？

这时我们要衡量的已经不是“某条 track 的分数”，而是框架本身的研究能力：

- 会不会搜索
- 会不会搭环境
- 会不会先做便宜基线
- 会不会留下中间证据
- 卡住后会不会换方向
- 新方向能不能真的变成可跑实验

所以，如果问题从“当前主线再提一点分”变成“框架从零开始能不能完成新任务”，那么 `program` 也必须变。

## Benchmark Carnese 的目标

`Benchmark Carnese` 不直接追主线 SOTA。

它的目标是建立一套**可重复、可隔离、可打包、可横向比较版本差异**的基准台，用来衡量：

- AutoResearch 版本 A 和版本 B，谁更会做从零开始的研究
- Planner / Thinker / Materializer / Worker 的链路是不是通的
- 系统是不是只会在固定轨道上调参
- 系统有没有真正的“搜索新方向并落实”的能力

它的核心不是“哪条模型最后分更高”，而是：

- **从零开始的研究效率**
- **从搜索到落实的闭环质量**
- **卡住时的换向质量**
- **整个过程的可回放性**

## 新问题定义

这次我们不是把问题定义成“脑电预测连续关节角”。

这次更合理的问题定义是：

> 先验证脑电是不是更适合稳定解出更粗粒度的运动状态，而不是连续精细轨迹。

在这个思路下，第一条 benchmark 选“步态划分”是合理的，因为它天然更适合作为从零开始的任务：

- 任务边界清楚
- 运动学侧可以先独立验证
- 不需要先假设脑电一定能还原精细连续值
- 更容易看出 Agent 是不是会先做基线、先做例外分析、先建评分脚本

也就是说，第一条任务 pack 可以是：

- 用 `RHTOE_z / RFTOE_z` 给每个试次划分支撑期和摆动期
- 先在运动学侧建立标签与评分
- 再观察框架能不能自己把这条任务推进下去

对应的任务定义已经单独写在：
[docs/2026-04-13_gait_phase_benchmark_from_zero.md](/Users/mac/Code/AutoBci/docs/2026-04-13_gait_phase_benchmark_from_zero.md)

## 我建议的新结构

我建议把 `Benchmark Carnese` 设计成三层，而不是只做一份新 prompt。

### 1. 任务包，也就是 task pack

这一层只定义任务本身，不带历史研究记忆。

它至少包含：

- 任务说明
- 数据边界
- 允许使用的输入
- 评分脚本
- 成功标准
- 不可碰规则

以步态划分为例，task pack 里应当只写：

- 用 `RHTOE_z / RFTOE_z` 给试次打支撑和摆动标签
- 按试次和阶段区间评分
- 先做运动学侧，不先假设脑电一定能成功

这一层不应该写：

- 当前 accepted stable best
- 当前最推荐模型家族
- 历史 topic 队列
- 之前哪条方向赢过

### 2. 框架包，也就是 framework pack

这一层定义 AutoResearch 在 benchmark 场景下怎么工作。

这一层才是新的 `program`。

它不回答“科学任务到底是什么”，而回答：

- 从零开始时系统的工作协议是什么
- 允许搜索到什么程度
- 什么时候应该先建基线
- 什么时候应该把搜索结果物化成可跑实验
- 卡住时什么时候要换方向
- 哪些中间过程必须留痕

也就是说，新的 `program` 不是旧主线的研究契约，而是 **framework benchmark 契约**。

### 3. 运行胶囊，也就是 run capsule

每跑一轮 benchmark，都生成一个完整胶囊，记录这一轮从零开始到底发生了什么。

它至少应该包含：

- 初始任务定义
- 搜索证据
- 规划判断
- 物化出来的新方向
- 实际命令和环境信息
- 评分结果
- 失败原因
- 最后的结论

这样每次跑完，你都能把这一轮完整打包、复盘和比较。

## 新的 `program` 应该定义什么

我建议不要去改现有 [tools/autoresearch/program.md](/Users/mac/Code/AutoBci/tools/autoresearch/program.md)。

而是并行新增一套 benchmark program。它的本体应当回答下面六件事。

### 1. 系统的目标是什么

新的 `program` 应当把目标改成：

> 在给定任务定义和固定边界的前提下，系统要尽可能自己完成从任务理解、环境准备、基线建立、方向搜索、实验物化到评分回写的闭环。

也就是说，裁判不再只是最终精度，而是**闭环完成质量**。

### 2. 已知输入是什么

新的 `program` 只能假设这些输入：

- 任务说明
- 数据边界
- 不可碰规则
- 评分脚本或评分协议
- 是否允许联网

不能默认存在：

- 既有 track manifest
- 历史最好结果
- 历史 topic 队列
- 当前优先方法族

### 3. 系统允许做什么

新的 `program` 应明确允许：

- 自己读任务定义
- 自己查仓库现有脚本和数据结构
- 自己联网搜索任务相关方法
- 自己提出新方向
- 自己补必要环境和中间文件
- 自己生成可跑实验

也就是说，**连新方向的环境都要让 Agent 自己配**，而不是只在预先写好的轨道里做小改动。

### 4. 系统必须留下什么留痕

新的 `program` 需要强制要求留痕，不然没法比较框架版本。

至少要有：

- 检索记录
- 工具调用统计
- 决策记录
- 物化记录
- 运行胶囊
- 失败原因
- 换向原因

否则最后只能看到“跑完了多少分”，看不到框架到底是怎么跑出来的。

### 5. 什么时候判定卡住

新的 `program` 里，卡住不再只是“分数没涨”。

在 benchmark 语境下，卡住至少包括：

- 只在旧方向里反复打转
- 有搜索但没有物化
- 有规划但没有 runnable 实验
- 有实验但总停在同一类失败

也就是说，这里的卡住是**研究行为层面的卡住**，不只是数值平台期。

### 6. 什么时候算完成

新的 `program` 里，一轮 benchmark 完成不等于“跑完时间预算”。

更合理的完成定义是：

- 已经至少形成一条可评分基线
- 已经至少尝试一次新方向物化
- 已经能够给出清楚结论：
  - 任务可行
  - 当前框架卡在哪
  - 下一步应该改框架还是改任务

## 衡量方式应该分成两层

我建议把 Carnese 的衡量分成两层，不能只看任务分数。

### 第一层：任务结果分

这是任务本身的分数。

对步态划分 benchmark，V1 可以看：

- 试次可用率
- 开始点和结束点误差
- 阶段区间重叠
- 例外覆盖率

这层回答的是：

- 这条任务本身有没有被做出来
- 结果质量够不够

### 第二层：框架行为分

这是更关键的一层。

它回答的是：

- 框架是不是会做研究，而不是只会执行固定轨道

我建议至少记录下面这些指标：

1. 环境建立成功率
   - 从零开始时，必要脚本、依赖、配置有没有补齐
2. 有效搜索率
   - 搜索是不是和任务相关，而不是泛泛搜一堆不落地的东西
3. 物化率
   - 搜到的新方向，有多少真的变成 runnable 实验
4. 首个可运行基线耗时
   - 多久能跑出第一条真正可评分基线
5. 卡住后的换向延迟
   - 卡住后多久会主动切方向
6. 空转率
   - 有多少轮只是重复旧思路，没有新物化
7. 工具成本
   - 搜索数、工具调用数、失败调用数
8. 胶囊完整度
   - 一轮跑完后，能不能完整回放这轮发生了什么

### 第三层：综合框架分

最后才做一个综合框架分，用来横向比较版本。

建议综合分不要只奖励最终结果，还要奖励：

- 更快地产生基线
- 更高的物化率
- 更低的空转率
- 更完整的过程留痕

一句话说：

- 任务结果分衡量“做得对不对”
- 框架行为分衡量“研究做得好不好”

## 我的方案

### 方案核心

我建议把 `Benchmark Carnese` 设计成一套**与主仓运行态隔离的 benchmark harness**。

它要做到：

- 可以从零开始反复重跑
- 可以打包每轮过程文件
- 可以比较不同框架版本
- 可以先接步态划分，再接别的任务

### 目录与职责建议

我建议的结构是：

- `benchmarks/carnese/tasks/`
  - 放不同任务包
- `benchmarks/carnese/framework/`
  - 放 benchmark program、流程协议、run capsule schema
- `benchmarks/carnese/runs/`
  - 放每次从零开始的完整运行胶囊
- `benchmarks/carnese/eval/`
  - 放任务评分和框架评分脚本

第一条任务包可以是：

- `benchmarks/carnese/tasks/gait_phase_v1/`

里面放：

- `task.md`
- `constraints.yaml`
- `score_task.py`
- `baseline_hints.md`（可选，默认不预加载）

### 运行原则

每次 benchmark run 都：

- 从干净上下文开始
- 不读主仓当前 topic 和记忆
- 只加载任务包和 benchmark program
- 生成独立 run capsule

这样你以后就可以一轮轮比较：

- 旧版框架
- 加了 planner 的框架
- 加了 thinker / materializer 的框架
- 加了自动换向和联网搜索的框架

### 为什么这是比“继续改当前架构”更好的路

因为当前主仓 AutoResearch 的目标是**推进正在研究的科学问题**。

而 Carnese 的目标是**评价研究框架本身**。

这两件事相关，但不是同一件事。

如果继续把它们揉在一起，你会很难回答：

- 这次进步到底来自更好的框架，还是来自旧主线已有知识
- 这次没进步到底是框架不行，还是任务本来就难
- 这次搜索能力看起来很强，是因为任务简单，还是因为框架真的更聪明

只有把 benchmark harness 单独隔离出来，这些问题才有机会被真正测清楚。

## 结论

### 要不要定义新的 `program`

要。

但不是替换当前主线 `program`，而是新增一套并行的 **benchmark program**。

### 新的 `program` 是什么

新的 `program` 不再是“固定主线里如何提分”的契约，而是：

> 在只有任务定义、固定数据边界和允许联网搜索的条件下，框架如何从零开始完成研究闭环的契约。

### 衡量方式是什么

必须分两层：

- 任务结果分
- 框架行为分

最后再合成综合框架分。

### 我的方案是什么

我的方案是：

1. 保留当前主仓 AutoResearch 作为正式研究系统
2. 另起一套隔离的 `Benchmark Carnese harness`
3. 用步态划分作为第一条任务包
4. 用 run capsule 记录每次从零开始的全过程
5. 用任务结果分 + 框架行为分来比较不同框架版本

这样我们后面终于能回答那个真正重要的问题：

**不是“这个模型最后多少分”，而是“这套 AutoResearch 框架到底会不会自己做研究”。**
