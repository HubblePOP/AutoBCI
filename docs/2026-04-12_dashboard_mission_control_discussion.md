# 8878 Dashboard / Mission Control 讨论帖

## 结论先说

这次讨论的重点不是“页面好不好看”，而是：

**AutoResearch 的控制面，首屏到底应该先回答什么问题。**

如果首页不能让人一眼看懂：

- 它现在在干什么
- 为什么这么做
- 下一步会做什么
- 我怎么接管

那它就还是一个“信息很多的看板”，不是一个可信的研究控制面。

这份讨论帖把三方方案并排放在一起：

1. `Codex`：当前对本项目上下文和运行态最熟的一版
2. `Claude Code`：从结构和维护性视角给出的评价
3. `Gemini`：从操作入口和视觉噪音视角给出的评价

目标不是投票，而是把：

- 已经形成的共识
- 还没达成一致的地方
- 需要你来拍板的点

收成一个清晰的讨论底稿。

---

## 我们现在到底在做什么

先把页面目标说清楚，不然后面所有分歧都会变成“审美之争”。

这页不是普通聊天页，也不是纯展示型 dashboard。  
它更像一个：

```text
Research Mission Control
```

也就是：

- 有任务
- 有队列
- 有当前运行
- 有结果
- 有证据
- 有人工接管点

所以首页最重要的不是“放多少东西”，而是**首屏优先级**。

我现在认为首屏只该优先服务下面这 4 个问题：

1. 当前在跑什么
2. 结果怎么样
3. 接下来会跑什么
4. 我怎么接管

其他东西都不是首屏同权信息。

## 三方方案

## 方案 A：Codex 版

### 核心判断

当前页面最大的问题不是某一个模块丑，而是：

- 首屏同权模块太多
- 内部 ID 占了主位
- 一句话信息单独占窗
- 次级面板没有降级
- 结果是“看起来很满，但没有重点”

### 设计原则

`less is more`

不是少做功能，而是：

- 首屏只放最关键的信息
- 次级内容默认折叠
- 语义名优先，内部 ID 降级
- 当前执行优先于研究全景

### 首页主张

首页应该收成：

```text
顶部状态条
左栏：Topic Inbox + Recommended Queue
右栏：Current Run + Task Console
全宽：Mainline Trend + Moonshot Scoreboard
折叠：Thinking / Research Tree / Artifacts / 方法摘要
```

### 对“什么最重要”的回答

最重要的是：

- 当前执行
- 主线趋势
- 接管能力

不是：

- 大量方法摘要
- 一堆内部运行名
- 展开的研究树和产物区

### 优点

- 最符合“控制面”定位
- 最符合你反复强调的“化繁为简”
- 能直接消掉当前大块空白和无效窗口

### 风险

- 首页会更克制，研究全景会被下沉
- 如果习惯“一屏看全”，会觉得信息变少了

---

## 方案 B：Claude Code 版

### 核心判断

Claude 抓到的重点是：

- 模块太多
- 层级不清
- 左右两栏嵌套混乱
- 一个大 HTML 里把操作区和展示区混成一锅
- inline CSS / JS 太大，维护成本高

### 设计原则

先把页面结构讲清楚，再谈美化和维护。

它最强调的是：

- 语义分区
- 结构注释
- 操作区 / 展示区明确拆开
- CSS 变量化
- CSS / JS 拆文件

### 首页主张

Claude 没给出一套完整的新首页信息架构，但它明确建议：

- 把操作区和展示区分开
- 把结构做清楚
- 不要再让整个页面像一锅大拼盘

### 对“什么最重要”的回答

最重要的是：

- 页面结构是否可理解
- 模块职责是否清晰

### 优点

- 抓住了结构债
- 对长期维护很重要
- 对后续继续演进非常必要

### 风险

- 它更像“重构视角”，不是“首页决策视角”
- 如果直接按这条走，很容易先花力气拆文件、抽变量，但首屏体验还是没有真正变简单

## 方案 C：Gemini 版

### 核心判断

Gemini 抓到的重点是：

- 背景噪音太重
- 核心操作区被埋没
- `Task Console` 应该更靠前

### 设计原则

让用户一进来就知道：

- 该从哪下手
- 核心操作入口在哪

### 首页主张

它的直觉更偏向：

- 清理背景
- 把 `Task Console` 提到更前面
- 把页面从“展示墙”改成“操作台”

### 对“什么最重要”的回答

最重要的是：

- 操作入口
- 页面第一眼的可读性

### 优点

- 抓到了“入口感”问题
- 站在用户“进来就要知道怎么用”的角度

### 风险

- 容易把问题过度归因到视觉噪音
- 如果只把 `Task Console` 抬上来，但不处理信息分层，页面还是会乱
- 如果把输入框做得比“当前执行状态”还更显眼，顺序可能会反

## 当前共识

下面这些点，三方其实已经高度接近：

### 1. 现在的首页太满了

这一点没有争议。

问题不是“功能太多”，而是：

- 同权模块太多
- 首屏没有优先级
- 一堆不该上首屏的东西上了首屏

### 2. 内部 ID 不该占首页主位

像这些信息：

- `overnight-...`
- `moonshot_upper_bound_...`
- `walk_matched_v1_64clean_joints_...`

对排障有用，但对首页没信息密度。

首页应该优先显示中文语义，例如：

- 这轮在冲什么目标
- 当前方法是哪类
- 当前是同试次还是跨试次
- 当前处于 smoke 还是 formal

### 3. 操作区必须更清楚

用户必须一眼知道：

- 当前任务
- 下一步
- 按哪个按钮接管

这说明 `Task Console` 不能再埋在深处。

### 4. 研究树 / 产物 / 思考链路不该首屏同权常驻

这些内容不是不重要，而是不该和“当前执行”抢首屏注意力。

它们更适合：

- 折叠
- tab
- 第二屏

### 5. 当前问题不是先拆文件

`index.html` 很大、inline CSS/JS 很重，这是真的。  
但这不是首页体验的第一病根。

更急的是：

- 把首页结构收对
- 把首屏信息层级排对

文件拆分应该放第二阶段。

## 还没达成一致的地方

真正还没完全锁死的，不是大方向，而是**首页主视觉到底怎么排**。

### 分歧 1：`Task Console` 到底是不是首页第一主块

一种思路是：

- 先看输入框和按钮

另一种思路是：

- 先看当前 run 和结果
- 再看如何接管

我当前更偏第二种。  
也就是：

```text
先看当前执行
再看怎么接管
```

而不是：

```text
先看输入框
再看系统现在做了什么
```

### 分歧 2：`Mainline Trend` 和 `Moonshot Scoreboard` 是否要继续分成两个大块

现在我更倾向于把它们合成一个全宽分析区：

- 上半是主线趋势
- 下半是纯脑电冲刺榜

原因是：

- `Mainline Trend` 是主信息
- 它不该再被其他次级 panel 挤压

### 分歧 3：首页还要不要保留方法摘要

`Algorithm Family Bests / Method Summary` 这块的信息量有用，但首页权重太高。

这里有两个方向：

- 从首页移走，进折叠上下文
- 继续留首页，但大幅压缩

我当前更倾向于：

- **从首页移走**

因为它正是造成“中间偏下面一大块空白”和“信息很多但不知所云”的原因之一。

### 分歧 4：背景要不要再继续减

我认为背景可以更克制，但它不是核心矛盾。  
核心还是信息架构。

如果后面还要继续调背景，我建议它只能是：

- 顺手减噪

不能变成这轮首页改造的主轴。

真正需要你决定的，只有下面 4 个。

### 决策 1：首页第一屏到底服务谁

## 最短总结

这次讨论真正要回答的问题不是：

> 这个 dashboard 漂不漂亮

而是：

> 对一个自动研究控制面来说，什么信息值得占首页主位？

我当前的结论是：

- 首页先服务“当前执行”
- 趋势图和纯脑电榜单是唯一值得保留的主分析区
- 研究树、产物、思考链路都应该降到折叠上下文
- 内部 ID 和方法摘要不能再抢首屏

如果沿着这个方向收，页面会第一次真正像一个：

```text
可信、可接管、可解释的 AutoResearch 控制面
```

而不是一个把所有东西都摊开的总览页。

## Claude Code 补充意见（经项目负责人确认同意）

日期：2026-04-12

### 趋势图的核心问题

当前趋势图的问题不是"用什么图形"，而是**它试图在一张图里同时回答太多问题**：主线长期趋势、每个算法族的表现、支线和参考线对比、停滞检测。

具体问题：

1. **算法族折线不应该存在** — 把同一算法族的所有尝试（smoke + formal）用 `buildLinePath` 折线连起来，产生剧烈上下波动（"心电图效应"），毫无信息量。
2. **阶梯线本身是对的** — running best 确实是阶梯函数，`buildStepPath` 没问题。
3. **散点本身足够** — 散点密度已经能反映"在某段时间密集尝试但没突破"的模式，不需要再画折线。
4. **停滞检测太简单** — 当前只检查最近 5 个 SOTA 步改进幅度 < 0.002，缺少时间维度和资源消耗维度。

### 建议方案：从"一张大图"改成"三层信息"

#### 第一层：健康指示器（首屏最重要的位置）

不是图表，而是一个简洁的状态句：

```text
主线 r = 0.4821  ▲ +0.0034 (24h)  |  连续 3 天无突破  |  最近 12 次尝试中 0 次刷新
```

这一行就能回答"现在怎么样"和"有没有在进步"。连续 N 天没突破或者最近 M 次尝试全部被丢弃时，直接变成警告色。这比任何图表都更快传递信息。

加一个**效率指标**：

```text
最近 24h: 尝试 8 次 / 突破 0 次 / token 消耗 ~42k
```

如果突破率长期为 0，人看到这行数字就知道该接管了。

#### 第二层：主线阶梯图（简化版）

只保留两样东西：

- 阶梯线（running best）
- 散点，但**不要连线**（去掉所有 `algorithm_series` 折线）

额外加一个**时间热力条**在图下方：一个窄横条，每天一个色块，颜色深浅代表那天做了多少次实验。"烧 token 但没进步"的时间段一目了然——色块很深（做了很多次）但阶梯线是平的。

#### 第三层：算法族对比（折叠，不占首屏）

不要用折线，改成**分组条形图**，每个算法族只显示它的**最佳成绩**。一眼看出 Ridge 最好是多少、XGBoost 最好是多少、LSTM 最好是多少。

### smoke vs formal 的区分

当前图把 smoke 和 formal 的结果混在一起画，建议：

- 散点用**不同形状**区分（圆 = formal，菱形 = smoke）
- 阶梯线只连接 formal 结果（只有 formal 才能真正"进主线"）
- smoke 的点保留（说明"我们试过了"），但不参与阶梯计算

### 对首屏信息架构的表态

讨论帖中 4 个决策，Claude Code 与项目负责人均选择：

1. 决策 1 → **A**（首屏先服务"当前执行和接管"）
2. 决策 2 → **C**（Task Console 和 Current Run 合并成连续的"当前执行区"）
3. 决策 3 → **A**（方法摘要从首页移走）
4. 决策 4 → **A**（底部三块默认折叠，tab 切换）

### 宽屏布局的补充

当前 `max-width: 1360px; margin: 0 auto` 在宽屏下左右对称居中，左栏（Topic Inbox + Queue）内容经常为空导致大面积空白。建议：

- 宽屏下改成**顶部窄条 + 下方全宽**的布局
- 或让左栏在没内容时自动收窄/隐藏

### 最短总结

趋势图最大的改进不是换图形类型，而是：

1. 把"一眼能看出状态"从图里提出来变成**文字指标行**
2. 图本身做减法——去掉算法族折线，只留散点 + 阶梯
3. 加一个**时间热力条**暴露"密集尝试但没进步"的时间段
4. "效率指标"（尝试次数 / 突破次数）直接用数字说话

## Claude Code 补充背景记录（项目负责人认可并要求纳入）

日期：2026-04-12

### 这次架构反思是怎么来的

这部分不是抽象设想，而是基于最近一次真实使用历史补的背景记录。

最开始我们曾考虑让 **Hermes Agent 充当整个 AutoResearch 的总控**。原因很直接：当前 Agent Loop 是沿着 Codex SDK 的运行方式搭起来的，它天然更像一个“在既有框架轨道里做局部推进”的系统，于是直觉上希望用 Hermes 负责更上层的搜索、调度和方向切换。

后来在一次睡前讨论里，我们开始追问：

> 为什么指标已经这么久没有提升了？

当时 Hermes 做了搜索，也给出了几种可能的新方案，并把内容写进了 Hermes 的 research tree。项目负责人随后又出差了一天。等两天后回来看，发现**指标基本没有变化**，真正出现的唯一“突破”是一个已经被明确点名要求检查的方向：去看运动学本身有没有上限空间，最后跑出了“纯运动学远高于脑电”的结果。

这个结果本身有价值，但它也暴露了更重要的问题：

- 不是 Hermes 没有搜索
- 不是 Hermes 没有写下候选方向
- 而是这些新方向并没有稳定进入真正的**可执行实验层**

也就是说，问题的主因更像是**架构缺口**，而不是 Hermes 本身“不够聪明”。

### 当前共识：问题主要是架构，不是单一 agent 的能力

Claude Code 给出的判断，项目负责人明确认可：

- 这件事主要不是 Hermes “不会想”
- 也不主要是 Codex “不会做”
- 更大的问题是系统当前更像一个“围绕既定主线做小步优化”的执行框架，而不是一个“把新论文、新方向稳定翻译成新实验”的研究控制面

所以后续才会自然演化出下面这个思路：

- 保留 `Worker` 负责真正执行
- 在其上层明确拆出 `Thinker / Planner / Materializer / Judgment`
- 让“新方向发现 → 队列判断 → 物化成 track → 真正起 run”的链路可见、可审计、可追责

### 为什么前端必须看见这些非 Worker 智能体

过去我们最困惑的地方，不是“模型为什么分低”，而是：

> 我明明已经告诉了方向，为什么系统过了两天还是没做？

如果前端只能看到 `Worker` 在跑什么，却看不到：

- `Thinker` 有没有真的形成新的假设
- `Planner` 有没有把它变成优先级判断
- `Materializer` 有没有把它落成新的 runnable track
- `Judgment` 有没有决定继续当前主线、还是降权老方向并触发孵化

那人只能看到一个“看起来很忙”的外表，却无法判断它到底是在推进，还是只是在旧轨道里打转。

这也是为什么后面会冒出一个自然问题：

> 要不要做一个类似 Cherry Studio 的壳，把这些调度智能体的思路、工具调用、任务优先级都显示出来？

### 当前结论：不要做“可视化思维链编辑器”，要做“结构化决策审计面板”

Claude Code 的建议，项目负责人认可并要求纳入这份讨论记录：

- 不建议照搬 Cherry Studio / Dify / Langflow 那种“工作流节点编辑器”式前端
- AutoResearch 需要的不是“把思维链全文 dump 给人看”
- 真正需要的是：**看到它做了什么判断、为什么这样判断、最后有没有真的落地成实验**

因此 dashboard 应该优先展示的不是原始推理文本，而是结构化链路：

```text
Topic Inbox → Retrieval → Decision → Queue Materializer → Worker → Judgment
```

每一段都应该有明确的“最近一次真实产物”：

- Topic Inbox：当前待处理的新方向数量、标题、优先级
- Retrieval：最近一次检索时间、证据条数、当前问题陈述
- Decision：最近一次推荐队列、降权理由、为什么该换向
- Materializer：有没有真的生成新的 track / thread / run_id
- Worker：当前在跑哪条轨、是 smoke 还是 formal
- Judgment：最近一次判断结果、下一步建议动作

### 对“思考可视化”的立场

这里的“看到它在思考”不等于“展示完整思维链”。

当前更合理的做法是：

- 首屏展示一条紧凑的**决策流水线状态条**
- 下方提供一个默认折叠的**思考时间线 / 决策时间线**
- 时间线只展示真实 packet 和结构化摘要，不展示原始冗长推理文本

换句话说，应该让人看到的是：

- 它有没有检索
- 有没有形成新判断
- 有没有真的把判断物化成新实验
- 如果没有，卡在哪一环

而不是让人去读一长串 agent 的自言自语。

### 这段背景对本次 dashboard 改造的直接影响

因此，这次 dashboard 改造并不只是“界面美化”，而是在补下面这件事：

> 让人一眼分清 AutoResearch 现在是在推进、在思考、在孵化新方向，还是只是在旧主线里原地忙碌。

这也是为什么本轮讨论最终会落到：

- mission control 首屏必须服务“当前执行和接管”
- 需要显式显示 `真推进`
- 需要显式显示 `卡住原因`
- 需要显式显示 `新方向孵化`
- 需要显式显示 `搜索与工具预算`
- 需要把 `Thinker / Planner / Materializer / Judgment` 的真实留痕带到前端

### 框架调度 benchmark

新增 `scripts/benchmark_framework_scheduling.py`，用来量化**框架调度效率**——不是模型精度，而是框架本身发现新方向、检测停滞、避免在旧算法里空转的能力。

当前真实数据（376 条 ledger 记录）的基线结果：

| 指标 | 值 | 含义 |
|------|-----|------|
| 多样性指数 | 0.70 | XGBoost 占 44%，分布偏重 |
| 突破率 | 3.08% | 每 ~32 次尝试才有 1 次 SOTA 刷新 |
| 最大连续无突破 | 98 次 | 曾经连续 98 次迭代没有进步 |
| 回滚率 | ~81% | 大部分修改最终被回退 |
| on-track 率 | ~48% | 约一半的修改判定为"在目标轨道上" |

### 对 Codex 端到端自动化计划的评审

**总体判断：方向正确，计划过重。**

Codex 的事实调查非常扎实——确认了当前系统的真实缺口（think 能判断停滞，但 incubate → materialize → launch 的链路没有端到端接通）。

**问题：**

1. **一次性补太多层**：停滞触发器 + incubate-topic CLI + 孵化模板注册表 + supervise_mission 升级 + dashboard 可观测性 + 完整测试 + 24h 试运行，任何一层出问题都会阻塞。

2. **停滞触发条件过复杂**：4 条 AND 逻辑（24h 无正信号 + 3 个 packet 队列头不变 + 2 条主轨耗尽 + 检索有增长但无新 smoke）容易导致**永远不触发**。

3. **模板注册表不必要**：V1 只有 3 个固定候选，硬编码列表就够了。

4. **最短路径更短**：用已有 `health_indicator.stagnation_level` 做触发 → 硬编码 3 个候选 → `supervise_mission` 加一个 `if stagnation → incubate` 分支 → 复用现有 campaign launch。

**建议：拆成两步**

- 第一步：最小端到端闭环（不做配置层、不做 CLI 入口、不做注册表）
- 第二步：跑通后收紧触发条件、候选列表改可配置、补完整测试

项目负责人已确认同意此评审意见。
