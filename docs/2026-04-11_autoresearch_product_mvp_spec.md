# AutoResearch 产品化 MVP Spec

## 一句话定义

AutoResearch 不是一个只服务 BCI 的实验助手，而是一个面向任何“目标清楚、可量化、可验证”的研究任务的自动研究操作台。

## 产品判断

1. 产品的核心不是卖某个单一行业，而是卖“把研究任务从想法推进到可执行实验”的自动化能力。
2. BCI 只是第一批验证场景，不应该把产品概念、界面和术语永久绑死在 BCI 上。
3. 产品默认面向“能定义目标、能定义约束、能定义评价指标”的研究型问题，而不是开放式创意任务。
4. 用户输入的第一件事应该是“我要解决什么量化问题”，而不是“我要用什么模型”。
5. 系统要把自然语言实验描述整理成结构化 research 计划，而不是只回复一段建议文字。
6. research 计划要能继续拆成方向、优先级、对照线、门禁、执行顺序和验收标准。
7. 产品要区分研究规划、代码执行、实验监督、结果总结四种角色，不能把它们混成一个聊天窗口。
8. 底层控制面必须独立存在，网页、CLI、Hermes、Codex 都只读写同一套状态真源。
9. 模型后端必须可切换，不能默认绑定 GPT，要支持 OpenAI、GLM、Kimi 等不同供应商。
10. 模型切换要分层，聊天模型、规划模型、代码模型、巡检模型可以分别配置。
11. 产品要支持“客户描述实验 -> 系统生成研究计划 -> 系统执行实验 -> 系统回写判断”的完整闭环。
12. 执行层必须有隔离环境能力，至少支持独立工作树、独立虚拟环境和可复用数据缓存。
13. 每次执行都要留下审计记录，至少说清楚做了什么、改了什么、跑了什么、结果怎样。
14. 运行态记忆和长期研究记忆必须分开，但两者之间必须自动写回和同步。
15. 长期研究记忆不能只是日志，它必须持续更新“当前关键问题、当前主线、当前护栏”。
16. dashboard 不只是看板，而要成为产品界面的一部分，而不是独立工程调试页。
17. MVP 的网页形态默认是聊天主控台，监控和实验状态嵌在同一产品壳里。
18. 网页聊天区第一版就要支持研究级能力，而不只是发几个控制命令。
19. 用户要能在网页里看到最近做了什么、正在做什么、接下来准备做什么。
20. 用户要能在网页里直接发起搜索新方向、写代码、跑 smoke、跑 formal 这类研究任务。
21. 图表层既要显示算法家族，也要显示方法层，避免“只看到大类、看不到具体尝试”。
22. 产品要能自动维护方向标签、算法家族、方法变体和控制实验的统一语义。
23. 新模型或新方法进入系统后，要能自动出现在图例、排行榜和研究摘要里。
24. 产品要支持排行榜，至少能按“模型 - 方法 - 分数”列出最近结果和历史最高结果。
25. 自动改主线必须有硬门禁，控制实验或不可信高分不能自动篡位。
26. 系统必须支持 bounded loop，也就是单轮会收口，但外层会自动续跑，不允许因为达到回合上限就停摆。
27. 网页里必须显式展示 agent 生命周期状态，例如 `searching / coding / validating / running / failed`。
28. 用户必须能在网页里暂停、恢复、终止、重排执行队列，而不是只能看不能控。
29. 产品必须支持多项目、多客户、多领域配置，不能把一个客户的术语、数据和策略污染到另一个客户。
30. 客户自己的实验术语、目标和成功标准，要能映射到系统内部统一的配置、指标和执行语义。
31. MVP 的成功标准不是功能最多，而是能把“一句研究需求”稳定推进到“一轮真实实验结果”。
32. 第一版要优先把控制面、研究计划、执行闭环和可视化做稳，再考虑通知、协作和商业包装。

## MVP 边界

1. MVP 先支持“目标清楚、数据可读、指标可算”的研究任务，不追求覆盖所有科研流程。
2. MVP 先支持单团队工作台，不把多租户权限系统作为第一阶段阻塞项。
3. MVP 先支持产品内统一控制面，不把 Hermes 继续当成业务内核。
4. MVP 先把网页、CLI 和状态真源打通，不要求第一版就覆盖所有外部聊天入口。
5. MVP 先让产品壳可用、可信、可监督，再决定哪些部分要进一步商品化或行业化。

## 当前结论

MVP 的产品形态可以继续推进，但技术实现已经不应再围绕“Hermes 持有业务内核”展开。
从现在起，产品层的唯一正确假设是：

- AutoBci control plane 是唯一真源
- 网页、CLI、Hermes、Codex 都是它的客户端
