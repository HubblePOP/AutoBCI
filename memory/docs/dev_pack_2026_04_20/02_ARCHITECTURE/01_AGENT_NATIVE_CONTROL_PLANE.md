# 单主控科研闭环设计

这个文件名保留给历史链接；当前定义已经收敛为单主控科研闭环。

AutoBCI 是科研闭环 APP / harness。它不模拟一个“AI 公司”，也不把产品主语放在多个角色 Agent 上。正确主语是：

```text
Program -> 研究方向 -> 执行沙盒 -> 固定评估 -> 结果复核 -> ledger
```

## 人类角色

owner 只管五件事：

1. 研究目的：这个任务为什么值得做。
2. 边界：哪些数据、文件、操作不能碰。
3. 预算：给多少时间、算力、token 和搜索空间。
4. 晋级：哪个结果进入当前最可信结果。
5. 意义：结果是否解释得通，是否值得继续。

## 系统职责

- Program 固定任务和边界，防止研究方向漂移。
- State 记录当前任务、当前方向和运行状态。
- Ledger 记录每一步动作、证据、结果和判断。
- Human Gate 在大步骤和高风险工具调用前停住。
- Sandbox 限制外部 coding agent 只能改允许文件。
- Evaluator 固定评估方式，避免把投机分数当成算法提升。

## 外部模型的位置

外部模型、Codex、Claude Code、Pi、MiMo、MiniMax、OpenAI-compatible provider 都只是工具或执行能力。它们可以提出建议、改结构文件、总结证据，但不能成为 AutoBCI 的科研状态真源。

## 反模式

不要把 AutoBCI 画成 PM、架构师、开发、测试、审稿人互相传文件的虚拟组织。这个模式会制造假边界，让上下文在角色交接里死亡。

如果需要并行搜索，可以开多个相同性质的探索分支；结果必须回到同一个 Program、State、Ledger 和结果复核流程里。
